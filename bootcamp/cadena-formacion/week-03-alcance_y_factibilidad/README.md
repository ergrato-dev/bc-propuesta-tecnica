# 📚 Semana 03 — Definir qué SÍ y qué NO entra al Proyecto

> **Versión**: Cadena de Formación · Técnico en Programación → Tecnólogo ADSO

---

## 🎯 Objetivos de Aprendizaje

Al finalizar esta semana, el aprendiz será capaz de:

1. Definir el alcance de un proyecto de software con precisión y justificación
2. Distinguir entre lo que está dentro del alcance y lo que queda explícitamente fuera
3. Construir una lista de entregables estructurada como base de la propuesta técnica
4. Evaluar la factibilidad técnica y económica básica de un proyecto
5. Documentar el alcance en el formato estándar de la propuesta técnica
6. Usar los requisitos de la semana 2 como insumo para definir el alcance

> Se conecta directamente con los requisitos levantados en S02 — solo entran al alcance los
> requisitos que el proyecto puede cumplir con los recursos disponibles.

## 🎓 Conceptos Clave

| Término | Definición breve |
|---------|-----------------|
| **Alcance** | Límites de lo que el proyecto SÍ incluye — entregables, funcionalidades y trabajo acordado |
| **In scope** | Dentro del alcance — el proyecto lo incluye y lo entrega |
| **Out of scope** | Fuera del alcance — el proyecto NO lo incluye en esta versión |
| **Entregable** | Resultado concreto y verificable que produce el proyecto |
| **WBS** | Work Breakdown Structure — descomposición jerárquica del trabajo del proyecto |
| **EDT** | Estructura de Desglose del Trabajo — nombre en español del WBS |
| **Factibilidad técnica** | Evaluación de si el equipo tiene la capacidad técnica para construir el sistema |
| **Factibilidad económica** | Evaluación de si el proyecto tiene sentido financiero (costo vs. beneficio) |
| **Restricción** | Condición fija que limita el proyecto (presupuesto máximo, fecha de entrega, tecnología obligatoria) |
| **Supuesto** | Condición que se asume verdadera para que el plan funcione, pero que puede cambiar |
| **Scope creep** | Crecimiento descontrolado del alcance cuando se agregan funcionalidades no planeadas |
| **MVP** | Minimum Viable Product — versión mínima del producto que entrega valor al cliente |

## ✅ Checklist de Verificación

Antes de entregar, verifica que tu documento cumple con:

- [ ] La declaración de alcance describe exactamente qué módulos o funcionalidades se construirán
- [ ] El out of scope lista al menos 3 cosas que el proyecto NO incluirá en esta versión, con justificación
- [ ] La lista de entregables incluye tanto software (módulos funcionales) como documentación
- [ ] La factibilidad técnica evalúa al menos: tecnología, equipo y tiempo disponible
- [ ] Se identificaron al menos 2 restricciones y 2 supuestos del proyecto
- [ ] El alcance es coherente con los requisitos del entregable S02 (no puede incluir cosas que no tenían requisito)
- [ ] El documento usa lenguaje formal y no deja secciones vacías
- [ ] El archivo está guardado como `entregable-s03-alcance.md` en tu repositorio

## 💡 Consejos para esta Semana

- **Más vale un alcance pequeño y cumplido que uno grande e incumplido.** Si dudas entre incluir o no una funcionalidad, ponla en el out of scope con la nota "posible fase 2".
- **El out of scope protege al consultor.** Si no lo escribes explícitamente, el cliente puede reclamar que "eso también lo habían pedido".
- **Usa tus requisitos S02 como base.** Solo puede entrar en el alcance lo que ya documentaste como requisito. Si descubres que falta un requisito, vuelve a S02 y agrégalo antes de cerrar S03.
- **La factibilidad no es opcional.** Antes de comprometerte a construir algo, debes verificar que tienes la capacidad técnica y el tiempo para hacerlo.
- **El caso FerreMax de esta semana muestra cómo Carlos pidió muchas cosas en la reunión — y el consultor tuvo que priorizar y dejar algunas para una versión 2.**
