# 📚 Semana 02 — Cómo Hablar con el Cliente y Documentar lo que Quiere

> **Versión**: Cadena de Formación · Técnico en Programación → Tecnólogo ADSO

---

## 🎯 Objetivos de Aprendizaje

Al finalizar esta semana, el aprendiz será capaz de:

1. Definir qué es un requisito y distinguir entre requisito funcional y no funcional
2. Aplicar al menos dos técnicas de levantamiento de requisitos (entrevista y cuestionario)
3. Preparar y conducir una entrevista estructurada con el cliente
4. Documentar los requisitos usando tablas con ID, descripción y prioridad
5. Identificar requisitos ambiguos, incompletos o contradictorios
6. Redactar un acta de reunión básica con los acuerdos y compromisos del cliente

## 🎓 Conceptos Clave

| Término | Definición breve |
|---------|-----------------|
| **Requisito** | Condición que el sistema debe cumplir para satisfacer la necesidad del cliente |
| **RF** | Requisito Funcional — qué debe *hacer* el sistema |
| **RNF** | Requisito No Funcional — cómo debe *comportarse* el sistema |
| **Levantamiento de requisitos** | Proceso de identificar, documentar y validar lo que el cliente necesita |
| **Entrevista** | Conversación estructurada con el cliente para descubrir sus necesidades |
| **Cuestionario** | Formulario con preguntas para recolectar información de varios usuarios |
| **Acta de reunión** | Documento que registra los acuerdos, decisiones y compromisos de una reunión |
| **Requisito ambiguo** | Requisito que puede interpretarse de más de una manera |
| **Prioridad** | Nivel de importancia de un requisito (Alta / Media / Baja) |
| **Stakeholder** | Persona afectada por el proyecto que puede tener requisitos propios |
| **Validación** | Proceso de confirmar con el cliente que los requisitos documentados son correctos |

## ✅ Checklist de Verificación

Antes de entregar, verifica que tu documento cumple con:

- [ ] Cada requisito funcional tiene un ID único (RF-001, RF-002, ...)
- [ ] Cada requisito no funcional tiene un ID único (RNF-001, RNF-002, ...)
- [ ] Los requisitos funcionales describen una *acción* del sistema (usa "El sistema debe...")
- [ ] Los requisitos no funcionales describen una *característica de calidad* (velocidad, seguridad, etc.)
- [ ] Hay mínimo 8 requisitos funcionales y 4 no funcionales
- [ ] Cada requisito tiene prioridad asignada (Alta / Media / Baja)
- [ ] El acta de reunión o evidencia del levantamiento está incluida
- [ ] Se identificaron requisitos que necesitan aclaración futura
- [ ] El documento usa lenguaje formal y los requisitos están escritos sin ambigüedad
- [ ] El archivo está guardado como `entregable-s02-requisitos.md` en tu repositorio

## 💡 Consejos para esta Semana

- **Habla con tu cliente real.** Esta semana es ideal para tener la primera reunión formal con el usuario de tu proyecto. Usa la plantilla de entrevista de la teoría.
- **Escribe los requisitos como el cliente los entiende, no como el programador los piensa.** "El sistema debe mostrar el inventario en tiempo real" es mejor que "Implementar WebSocket para sincronización en tiempo real".
- **Más es mejor al principio.** Documenta todo lo que el cliente menciona, aunque parezca obvio. En la semana 3 decidirás qué entra en el alcance y qué no.
- **La entrevista es una conversación, no un interrogatorio.** Empieza con preguntas abiertas ("¿Qué problemas tiene hoy?") y luego afina con preguntas específicas.
- **Usa el caso FerreMax como guía.** La segunda reunión con Carlos Mendoza es un ejemplo de cómo documentar requisitos de un caso real.
