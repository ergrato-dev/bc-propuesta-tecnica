# 📚 Semana 01 — ¿Qué es un Proyecto de Software y una Propuesta Técnica?

> **Versión**: Cadena de Formación · Técnico en Programación → Tecnólogo ADSO

---

## 🎯 Objetivos de Aprendizaje

Al finalizar esta semana, el aprendiz será capaz de:

1. Distinguir un proyecto de software de otras actividades de desarrollo
2. Identificar las fases del ciclo de vida básico de un proyecto
3. Reconocer los roles principales en un equipo de proyecto
4. Comprender qué es una propuesta técnica y para qué sirve
5. Diferenciar una cotización, una propuesta técnica y un contrato
6. Identificar los componentes principales de una propuesta técnica profesional

## 📌 Entregable de la Semana

**Documento:** `Contexto y Justificación del Proyecto` (Sección 1 de tu propuesta técnica)

El aprendiz completa la plantilla `3-proyecto/starter/entregable-s01-contexto-proyecto.md` con información de su **proyecto real aprobado**, cubriendo:

- Descripción del problema que resuelve
- Identificación del cliente / usuario principal
- Justificación del proyecto
- Idea general de la solución
- Listado inicial de stakeholders

> 📁 Este documento es la **primera sección** de la propuesta técnica que se construirá
> a lo largo de las 9 semanas. Guárdalo en tu repositorio personal de GitHub.

## 🎓 Conceptos Clave

| Término | Definición breve |
|---------|-----------------|
| **Proyecto** | Esfuerzo temporal con inicio, fin y resultado único |
| **Ciclo de vida** | Fases que atraviesa un proyecto de principio a fin |
| **Stakeholder** | Persona o grupo afectado por el proyecto |
| **Propuesta técnica** | Documento que describe CÓMO se hará el proyecto |
| **Cotización** | Documento con precios y condiciones de venta |
| **PM** | Project Manager — responsable de la gestión del proyecto |
| **Sponsor** | Quien financia y autoriza el proyecto |
| **Entregable** | Resultado concreto producido por el proyecto |
| **Alcance** | Límites de lo que el proyecto SÍ incluye |
| **Requisito** | Condición que debe cumplir el sistema o el servicio |

## ✅ Checklist de Verificación

Antes de entregar, verifica que tu documento cumple con:

- [ ] El nombre del proyecto está claramente definido
- [ ] El problema está descrito en términos del cliente (no técnico)
- [ ] El cliente/usuario está identificado con nombre y rol
- [ ] La justificación responde "¿por qué es necesario este proyecto?"
- [ ] La idea de solución es comprensible para alguien sin conocimientos técnicos
- [ ] Se listaron mínimo 3 stakeholders con su rol en el proyecto
- [ ] El documento usa lenguaje formal (sin jerga informal, sin abreviaciones sin definir)
- [ ] El archivo está guardado como `entregable-s01-contexto-proyecto.md` en tu repositorio

## 💡 Consejos para esta Semana

- **No te preocupes por la perfección.** Esta es la primera sección de un documento que irás mejorando semana a semana.
- **Habla con tu instructor** si no tienes claro cuál es tu proyecto asignado o si tu idea no está aprobada aún.
- **Usa el caso FerreMax como referencia** cuando no sepas cómo redactar algo — los ejemplos en teoría y práctica están basados en él.
- **Guarda TODO en GitHub** desde el primer día — el historial de cambios es parte de tu evidencia de aprendizaje.
- **Tip de redacción**: Escribe el problema como si se lo explicaras a un familiar sin conocimientos técnicos. Si lo entiende, vas bien.
